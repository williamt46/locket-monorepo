New canonical component — confirmed from prototype + screenshots

Phase Segmented Bar

This is Locket's most visually distinctive UI element. A full-width grid of pill-shaped segments, one per cycle day, color-coded by phase. Gap: 2px between segments, height: 10px, border-radius: 999px. Future/unknown days use #F2F2F7 (Pale Lavender Mist). The grid uses CSS grid-template-columns: repeat(N, 1fr) where N is the cycle length. This component does not exist in design.md — it was discovered from the real screens and confirmed in CycleTrendsScreen.html source.

New confirmed component — Tab Navigation

Two tab styles are in production simultaneously

Underline tabs (Cycle Trends, CycleTrendsScreen.html): active tab gets Locket Blue text + 2px bottom border. Pill tabs (Share Access — all three states): active tab gets Locket Blue filled pill with white text. Neither style is in design.md. Both are real and established — they appear to serve different contexts (top-level navigation vs. sub-section switching) but this isn't documented anywhere.

What Changed From Round 1

Several conflicts from the first review are either resolved or reframed by the real screens.

New conflict (critical) — A third primary blue

The prototype runs MD3's primary: #00569D alongside locket-blue: #006EC7

The CycleTrendsScreen.html Tailwind config declares both. #00569D is the MD3 primary container color — it's 17% darker than Locket Blue and is used for text selection highlight (selection:bg-primary/20). On screen they appear close enough to be indistinguishable, but they're different tokens with different semantic roles. If MD3's primary gets applied to UI elements that should use locket-blue, the blues will read as inconsistent. The nav underline in CycleTrendsScreen.html uses border-locket-blue correctly, but the MD3 token could contaminate other components silently.

New conflict — Public Sans used in prototype, not in mobile app code

Three different font choices across three surfaces

design.md specifies SF Pro Display (iOS heading). typography.ts uses Avenir Next (heading) / Helvetica Neue (body). CycleTrendsScreen.html loads Public Sans from Google Fonts for all three type roles. All three are clean sans-serifs, so visual divergence is subtle — but they are different characters. Public Sans is the US federal government typeface: authoritative, legible, slightly institutional. Avenir Next is a geometric with warm curves. They're not interchangeable design choices. The landing page will need a decision on which face to use, and that choice will cascade.

Resolved differently — Phase card pattern

Real app uses full-tint card; code uses white + left border; both differ from design.md

Round 1 flagged that PhaseInsightCard.tsx uses borderLeftWidth: 3, backgroundColor: '#FFFFFF'. The real Cycle Insights screenshot shows the card with a full teal tint background (not white). The CycleTrendsScreen.html uses white cards with tinted icon badges — not tinted cards. So there are now three card variants in the real product and zero documentation of when to use which. The real screen is the most polished-looking; the code is furthest from it.

Conflict — Back navigation is text, not icon

Real screens use "← Back" text; design.md specifies arrow_back Material Symbol

Every real screenshot uses plain text "← Back" or just "‹" (iOS native back indicator). design.md specifies the arrow_back Material Symbol. The CycleTrendsScreen.html doesn't have a back button at all (it's a top-level tab screen). The "← Back" text pattern reads as an unstyled iOS WebView default, not a designed interaction. This is a visible inconsistency between the spec's intent and what shipped.

Conflict — Secondary text color

Prototype uses MD3's outline: #717783; design.md specifies #8E8E93

These are different grays. #717783 is slightly cooler and darker. #8E8E93 is the iOS system secondary label color. The prototype consistently uses text-outline (#717783) for metadata, dates, and descriptions. The mobile code inherits iOS defaults which would resolve to ~#8E8E93. On screen the difference is marginal, but it's another unresolved fork between the web prototype and the mobile native app.

Unspecced — Avatar system (Active Shares screen)

Circular letter avatars use arbitrary green + dark red — neither in design.md

The Active Shares screen shows circular avatars with initials: "DS" on a green circle, "DH" on a dark red circle. The green looks like #2A9D8F (arcticTeal) for DS but could be a generated color. The red for DH is not Terracotta (#D1495B) — it looks more like a generic avatar-generation hue. There's no documented avatar color generation logic. If this screen ships as a showcase, the arbitrary color assignment will be visible.

Unspecced — "Expired" badge uses Locket Blue tint for a negative state

The "Expired" status badge on Active Shares uses bg-[#E5F1FA] — the trust/action blue tint

Locket Blue and its tint connote trust, sovereignty, and active action throughout the system. Using it for "Expired" (a past/revoked state) creates a semantic collision — the expired state reads as active/positive rather than neutral/concluded. A neutral gray badge (#F2F2F7 background, #717783 text) would be more appropriate and is already in the palette.

Still missing — Dark mode in the native codebase

Zero dark mode tokens in colors.ts; the HTML prototype supports dark mode via class but it's unused

The CycleTrendsScreen.html has darkMode: "class" in its Tailwind config, but the <html> element uses class "light" — dark mode is wired but not activated. The React Native app has no dark mode tokens at all. design.md specifies a complete dual-mode system. This is still fully unimplemented.

Clarified — Icon style is more nuanced than "all FILL:1"

design.md says all icons use FILL:1; real prototype uses FILL:0 for UI icons, FILL:1 for semantic/phase icons

This is actually a better approach than the spec. UI icons (chevron_right, history, timeline) use outlined style (FILL:0) — they're structural. Phase and data icons (water_drop, psychiatry) use filled style (FILL:1) — they're expressive. The prototype makes this distinction explicitly via CSS classes .material-symbols-outlined and .material-symbols-fill. design.md should be updated to reflect this two-tier icon approach, not corrected.

What Would Close the Remaining Gaps

  • 1

    A decision on the font

    Public Sans (prototype), SF Pro (spec), Avenir Next / Helvetica Neue (native code) — pick one and document it. For the landing page specifically: if it's targeting APAC healthcare builders who will also see the portfolio, the font choice signals something. Public Sans reads institutional/civic. Inter reads technical/fintech. A custom choice (e.g. DM Sans, Instrument Sans) signals craft. Uploading a sample rendering of the hero headline in 2–3 candidate fonts would close this gap faster than arguing about it in the abstract.

  • 2

    IntegritySeal state spec

    This is a trust-facing, animated component that communicates data sovereignty status in real time. It currently runs on a private color system (gold, emerald, Tailwind blue, dark crimson) with zero connection to the design system. A single image showing the five states — even a rough sketch with labeled colors — would let it be rebuilt on-system. This matters for the landing page too: an animated IntegritySeal would be a natural hero element that's both technically credible and visually distinctive.

  • 3

    The empty state design (fix the emoji issue)

    The "No Pending Requests" empty state uses a 📥 system emoji for its icon — which breaks the Material Symbols consistency everywhere else and renders differently across OS versions. An upload of a designed empty state (using an appropriate Material Symbol at ~40–48px, or a custom illustration) would also clarify whether Locket uses illustration at all. Knowing that would directly inform whether the landing page should have illustration or rely purely on UI mockup screenshots and the phase bar as visual elements.

On-brand

The phase bar in the hero achieves something no other SaaS landing page does — it shows the actual product interface as a visual element rather than a device mockup. It conveys phase awareness, personal data ownership, and visual warmth simultaneously. The trust pills use real technical vocabulary (not "bank-level security"). Still generic

The "dark hero + centered headline + CTA row + social proof pills" structure is still the dominant SaaS landing page template. The phase bar breaks from convention but only within a familiar layout container. The headline "Your Private History, Set in Stone" is good but "Set in Stone" could read as inflexible rather than permanent/sovereign to readers outside the Ledger metaphor frame.

What would improve it

The phase bar in the hero should animate — each segment populating left-to-right as the page loads, revealing the cycle pattern over ~1.5 seconds. This would make it feel alive and data-driven rather than static decoration. Pairing the hero with a visible phone mockup showing the real Cycle Trends screen would ground the abstract bar in a familiar product moment.
