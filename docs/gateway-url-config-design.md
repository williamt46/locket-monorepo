# Design: Configurable Gateway Base URL for Device Builds

Branch: fix/restore-lazy-import-device

Mode: Engineering (internal)

## Problem Statement

The mobile app hardcodes the gateway base URL `http://localhost:3000/api` as a
local `const GATEWAY_URL` in three files:

- `apps/mobile/src/services/BlockchainService.ts:4` (used at :29 anchor, :66 batch, :94 verify)
- `apps/mobile/src/services/SyncService.ts:3` (used at :32 upload, :71 grant)
- `apps/mobile/src/screens/LogDataScreen.tsx:11` (used at :94 verify)

On the simulator `localhost` resolves to the dev machine, so it works. On a physical
device `localhost` is the phone itself, so every gateway `fetch` fails with
`Network request failed` — the exact symptom that opened this session (batch anchor
failures in `BlockchainService.ts:84` and `BackgroundSyncService.ts:152`). There is
no config mechanism today: plain `app.json`, no `app.config.ts`, no `eas.json`, no
`EXPO_PUBLIC_*` usage, no `extra` block.

## Goal & Scope

- **In scope (MVP → App Store review ready):** anchoring works on a physical device
  against a reachable gateway; the base URL is set at build time, not hardcoded.
- **Out of scope:** PRE / consent / sharing on device. Blocked on the unsolved iOS
  Hermes WASM / Umbral init failure; the gateway-side kFrag fallback is demo-only and
  must never ship (hard rule: secret key never leaves the device). Deferred, not solved.

## Premises (agreed this session)

1. A configurable URL is necessary but ~5% of "production." Production here means
   **App Store review ready**, not merely HTTPS. Agreed.
2. Anchoring-to-production is on the MVP timeline; sharing/PRE-to-production is not.
   PRE and consent remain unsolved and out of scope. Agreed.
3. Doing the config change now is worth it: it unblocks physical-device QA against the
   dev machine (LAN IP `192.168.0.164`) today and is the foundation the production
   build profiles sit on. Agreed.

## Approaches Considered

### Approach A: Single config module + `EXPO_PUBLIC_GATEWAY_URL` (CHOSEN)
One `apps/mobile/src/config/gateway.ts` exporting a `GATEWAY_URL` that reads
`process.env.EXPO_PUBLIC_GATEWAY_URL` and falls back to `http://localhost:3000/api`.
Replace the three hardcoded consts with imports. Effort S, Risk Low.

### Approach B: `app.config.ts` `extra` + `expo-constants` + `eas.json` profiles
Per-profile URLs via `Constants.expoConfig.extra`, with EAS build profiles
(dev/preview/production). Effort M, Risk Low-Med. Right move once a gateway is
actually deployed; premature now (no production environment to point at).

### Approach C: In-app `__DEV__`-only host switcher
Runtime base URL in AsyncStorage, editable from a dev-only screen. Effort M, Risk Med.
A QA convenience (change host on-device without rebuilding when the LAN IP shifts),
not a production path. Only ever as a `__DEV__`-gated add-on to A.

## Recommended Approach

**A now, B later.** A is a ~15-minute diff that unblocks device QA immediately and is
exactly what B builds on. Pull in B when a deployed HTTPS gateway exists. C is optional
and, if added, must be hard-stripped from production builds (a writable endpoint URL is
an attack surface in a health app).

### Implementation spec (Approach A)

1. Create `apps/mobile/src/config/gateway.ts`:
   ```ts
   // Base URL for the Locket gateway. Overridable at build time via
   // EXPO_PUBLIC_GATEWAY_URL (Metro bakes EXPO_PUBLIC_* at compile time — restart
   // with `expo start -c` after changing it). Defaults to localhost for the simulator.
   export const GATEWAY_URL =
     process.env.EXPO_PUBLIC_GATEWAY_URL ?? 'http://localhost:3000/api';
   ```
2. Replace the local `const GATEWAY_URL = 'http://localhost:3000/api'` in all three
   files (`BlockchainService.ts`, `SyncService.ts`, `LogDataScreen.tsx`) with
   `import { GATEWAY_URL } from '../config/gateway'` — all three sit one level under
   `src/`, so the relative path is identical for each.
3. Document `EXPO_PUBLIC_GATEWAY_URL` in `.env.example`. For device QA, set it in a
   local `apps/mobile/.env` file (gitignored) — NOT as an inline shell var. SDK 54
   resolves `EXPO_PUBLIC_*` through the generated `expo/virtual/env` module, populated
   by `@expo/env` from `.env` files; inline vars don't reach it, so the app silently
   falls back to localhost. Example: `EXPO_PUBLIC_GATEWAY_URL=http://192.168.0.164:3000/api`.
   Restart with `expo start -c` after changes. Verified via `expo export`: the bundle
   baked the LAN IP from `.env` (zero `localhost` in output).

## Success Criteria

- Grep shows `http://localhost:3000/api` in exactly one file (`config/gateway.ts`), as
  the fallback only.
- Simulator behavior unchanged (default resolves to localhost).
- On a physical device with `EXPO_PUBLIC_GATEWAY_URL` set to the dev machine's LAN IP
  (gateway + Fabric running), a batch anchor returns 201 and the SyncEngine logs
  success instead of `Network request failed`.
- No behavior change to any crypto/ledger path (URL plumbing only).

## Production Gates (out of scope here, tracked separately)

App Store review readiness for anchoring needs, beyond this config change:
1. **Deployed HTTPS gateway.** The gateway is welded to a local Fabric network
   (`localhost:7051`) + local MSP crypto material. "Deploy the gateway" means hosting
   Fabric (or a managed ledger), the gateway, TLS, and secrets. No deploy config exists
   today (no Dockerfile / vercel / fly).
2. **iOS App Transport Security.** Production requires HTTPS. `ios/locketapp/Info.plist`
   already sets `NSAllowsArbitraryLoads=false` + `NSAllowsLocalNetworking=true`, so dev
   device QA over http to a LAN IP may work; if ATS blocks the specific RFC-1918 host,
   add a scoped `NSExceptionDomains` entry (never `NSAllowsArbitraryLoads=true`).
3. **EAS build + submission pipeline.** `eas.json` profiles (Approach B) injecting the
   production URL; App Store Connect app record, signing, privacy nutrition labels
   (health data), review.
4. **App Review surface check.** With sharing/PRE out of scope, confirm no
   half-wired consent UI is reachable that would fail or mislead a reviewer.

These are a separate, larger design. This doc only covers making the URL configurable.

## Dependencies

- Gateway + Fabric running for device QA (both confirmed healthy this session:
  gateway on `:3000`, `POST /api/anchor/batch` → 201).
- No dependency on the PRE/WASM campaign — anchoring is a plain `fetch`.

## The Assignment

Before writing Approach B or any deploy config: **run the anchor flow once from a
physical device** against `EXPO_PUBLIC_GATEWAY_URL=http://192.168.0.164:3000/api` (with
gateway + Fabric up) and confirm a 201 + on-chain assetId. That single device round-trip
tells you whether the config change is the whole device story for anchoring, or whether
something else (ATS, network, cleartext-exception) also bites on hardware — cheaper to
learn now than after you've built EAS profiles on top.
