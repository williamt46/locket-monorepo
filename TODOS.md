A future merge pass would need an HMAC-hashed day key (plaintext `day:<date>:<event>` ids would leak the period calendar at rest — see §11 E3) plus a field-level merge policy honoring the no-convergence rule for note text (this file, 2026-07-22 entry). - [ ] **P3: Cycle factors → prediction baseline (Phase 3)** — Deferred by decision D3.4/ENG-D3. Pregnancy, lactation, and contraceptive spans from HealthKit are currently preserved as note text only. Giving them structured homes requires a factor-span model, prediction suppression over those spans in `PredictionEngine`, and reversing the deliberate Flo pregnancy-key drop in `ImportService.ts` for consistency. Greenfield — no factor concept exists in `BaselineCycleData`/`phaseBoundaries` today.
- [ ] **P2: Large imports block the JS thread with no progress** — `batchInscribe` (`apps/mobile/src/hooks/useLedger.ts`) awaits AES-GCM `encryptData` + `generateIntegrityHash` sequentially, one entry at a time. At ~1-3ms per pair a 5k-entry HealthKit import is roughly 10-30s of awaited work, and `commitPreview` is atomic so `HealthKitPreviewScreen` can only show an indeterminate "Sealing…" spinner (it sets `commitTotal` but has no progress source). The user may background or kill the app mid-commit. Fix: chunk the loop (~200 at a time), yield to a macrotask between chunks so the UI animates, and thread a progress callback through `commitPreview` → the screen's existing `commitTotal`/`commitDone` state. Flagged by the performance specialist during `/ship` 2026-07-23 (confidence 82). Only bites on large imports; a typical few-hundred-entry import is sub-second.
- [ ] **P2: Preview decrypts the entire ledger on every build** — `decryptExistingEntries` (`apps/mobile/src/hooks/useLedger.ts`) sequentially awaits `decryptData` for every record to build the collision date index, and re-runs on every preview production including retries. For ~1000 historical records that is ~1-3s before the preview can render, with no progress or cancel affordance. It is also more work than needed: collision detection only needs the day key, but the full plaintext of every historical entry is decrypted and held in memory. Fix options: cache the decrypted date index for the session (invalidate on write), decrypt in chunks with progress, or store an unencrypted day key on `StorageRecord` so collision detection needs no decryption at all and only colliding days get decrypted for display. **Note the privacy tradeoff on the last option** — a plaintext day key at rest is exactly the leak that killed deterministic ids (§11 E3 of the CEO plan), so prefer caching over denormalizing. Flagged by the performance specialist during `/ship` 2026-07-23 (confidence 80).
- [ ] **P2: Import Undo invalidates the sync epoch globally** — `purgeByIds` calls `BackgroundSyncService.invalidate()` before deleting, which is required (otherwise the anchor write-back from the same commit resurrects every purged record). But `invalidate()` is global: an UNRELATED in-flight anchor batch that already succeeded on-chain has its write-back discarded by the epoch guard, so those records keep `status: 'local'` and lose their `assetId` despite being anchored. The next sync re-anchors them, creating duplicate on-chain assets. Unlike nuke/superNuke the ledger survives here, so the divergence persists. Fix direction: give the sync engine a targeted cancel (drop only the purged ids from the pending write-back set) or await the in-flight sync before purging. Flagged by the security specialist during `/ship` 2026-07-24 (confidence 75).
- [ ] **P2: No escape from a hung commit on the HealthKit preview screen** — during `phase === 'committing'` the header Back button is disabled AND a `beforeRemove` listener blocks hardware back and the swipe gesture, so if `commitPreview` never settles the only way out is killing the app. The block is correct (leaving mid-commit strands the import with no Undo), but it needs a way out: race the commit against a watchdog (~60s) that surfaces "This is taking longer than expected" and re-enables leaving, or transitions to the existing `commitError` path. Flagged by the design specialist during `/ship` 2026-07-24.
- [ ] **P3: Screen-level fixes have no automated coverage** — the commit re-entrancy ref, the `beforeRemove` block, and the disabled Back button in `HealthKitPreviewScreen` are guarded by nothing but device QA; the repo has no React Native rendering infrastructure. The double-tap guard is the highest-risk of the three (a double-commit writes every day twice with only the second `CommitResult` surviving, putting half the duplicates beyond Undo). Either extract the commit sequencing into a testable non-React helper, or add explicit device-QA checklist items for rapid double-tap on Inscribe and swipe-back mid-commit.
- [ ] **P3: The non-ASCII truncation test does not discriminate byte-exact from UTF-16 counting** — `FileSystemLedger.test.ts`'s multibyte test truncates a trailing ASCII character, so the written length differs from BOTH `utf8ByteLength(serialized)` and `serialized.length`; a reverted UTF-16 guard would also fail it, just incidentally. The real hazard — a short write hiding INSIDE the multibyte slack, where surviving bytes happen to equal `serialized.length` — is untested. Fix: compute the slack explicitly and truncate into it, so the case passes under a `serialized.length` guard and fails only under the byte-exact one. Flagged by the testing specialist during `/ship` 2026-07-24.
- [ ] **P3: HealthKit truncation detection is dormant** — `ImportPreview.truncation` ships as tested-but-unused plumbing: HealthKit exposes no earliest-authorized-date API, so a partial-history grant cannot be detected programmatically. `createLibraryClient` omits the optional `getEarliestAuthorizedDate` hook and the field is always `null` in production. The user-facing mitigation is the preview's "Found {range}" header, which they compare against expectation. Re-check if Apple ever exposes the authorized window.
